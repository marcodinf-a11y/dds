# at-08d20126: Pipeline: Claude CLI Wrapper and Config

## Objective

Implement the thin wrapper around the `claude` CLI that all LLM operations depend on, along with the pipeline configuration schema and default config file. After this task, any module can invoke `claude -p` through a typed function that handles timeouts, retries, rate limits, inter-call delays, and ajv double-validation of responses.

## Context

No codebase exists yet for the pipeline engine. This task creates the foundational LLM invocation layer that every downstream task (Ring runners, fix functions, refinement, orchestration) imports. The `claude` CLI is invoked via `child_process.execSync` in synchronous mode because the refinement loop is inherently sequential. All configuration parameters have compiled-in defaults and are optionally overridden by `pipeline/config.json`, which is validated against its own JSON Schema at startup.

Key configuration parameters and defaults:
- `refinement.max_iterations`: 5
- `refinement.convergence_threshold`: 0.7
- `timeouts.ring1_check_seconds`: 60
- `timeouts.ring2_check_seconds`: 90
- `timeouts.fix_call_seconds`: 120
- `claude_cli.max_retries_on_short_429`: 3
- `claude_cli.backoff_multiplier`: 2
- `claude_cli.delay_between_calls_ms`: 2000

Rate limit handling: short 429 (retry-after <= 60s) retries with exponential backoff; long 429 (retry-after > 60s) aborts immediately.

## Approach

1. Create `src/schemas/pipeline-config.schema.json` with `$schema` set to `http://json-schema.org/draft-07/schema#`. Define the top-level object with `additionalProperties: false` and three optional nested objects: `refinement` (with `max_iterations` as integer and `convergence_threshold` as number between 0 and 1), `timeouts` (with `ring1_check_seconds`, `ring2_check_seconds`, `fix_call_seconds` as positive integers), and `claude_cli` (with `max_retries_on_short_429` as non-negative integer, `backoff_multiplier` as positive number, `delay_between_calls_ms` as non-negative integer). No required fields at any level.

2. Create `pipeline/config.json` as a minimal valid config file with an empty object `{}` (all defaults apply). Add a comment-like `_description` field explaining that all fields are optional.

3. Create `src/llm/claude-cli.ts` exporting:
   - A `PipelineConfig` interface reflecting the schema structure with all fields optional.
   - A `loadConfig()` function that reads `pipeline/config.json` if it exists, validates it against the schema using `ajv`, merges with compiled-in defaults, and returns a resolved config object. If the file does not exist, returns compiled-in defaults without error.
   - A `callClaude(prompt: string, jsonSchema: object, config: PipelineConfig, timeoutSeconds: number)` function that: (a) enforces `delay_between_calls_ms` since the last call using a module-level timestamp, (b) invokes `child_process.execSync` with `claude -p "${prompt}" --output-format json --json-schema '${schema}' --max-turns 1`, (c) parses the JSON output, (d) validates it with `ajv` against the provided schema, (e) returns the parsed object, (f) on short 429: retries with exponential backoff, (g) on long 429: throws an error that the pipeline interprets as abort.

## Constraints

- Use `child_process.execSync` only — no async execution.
- Do not import from any other `src/` modules (this is a leaf dependency).
- All config fields must have compiled-in defaults; the pipeline must work without `pipeline/config.json` existing.
- Do not add CLI argument parsing; this module is a library, not an entry point.
- Use `ajv` v8 for schema validation (Draft-07 mode).
