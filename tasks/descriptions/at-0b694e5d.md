# at-0b694e5d: Pipeline: Orchestration

## Objective

Implement the pipeline orchestrator that coordinates a full four-phase pipeline run from a root specification through validation, decomposition, cross-level checking, and reporting. After this task, the system can execute a complete pipeline producing validated atomic tasks or halting with escalation reports.

## Context

The refinement loop (at-353cb055) handles single-document validation. This task builds the cross-document orchestration layer on top. The orchestrator implements two main entry points: `runPipeline()` for full pipeline runs and `onSpecChange()` for incremental re-validation when a spec is modified.

The four sequential phases (REQ-13):
1. Validate the root spec through the refinement loop.
2. Decompose the spec into impl docs (via LLM generation prompt from Impl Doc 2), validate each through refinement.
3. Decompose each impl doc into atomic tasks (via LLM generation prompt from Impl Doc 3), validate each through refinement, check cross-task invariants.
4. Run all cross-level invariants (CL-S01 through CL-S04, CL-T01 through CL-T05, CL-F01, CL-F02) across the full document tree.

The pipeline halts at any phase where an escalation occurs. After successful decomposition and validation, parent documents are updated with child references and status set to `decomposed` (REQ-19).

## Approach

1. Create `src/pipeline/orchestrate.ts` exporting:
   - A `PipelineResult` interface with fields: `runId: string`, `rootSpecId: string`, `status: 'completed' | 'escalated' | 'aborted'`, `phase: number` (phase reached), `perDocumentResults: DocumentResult[]`, `crossLevelResults: CrossLevelResult[]`, `stats: PipelineStats`.
   - A `DocumentResult` interface with: `id: string`, `level: string`, `status: string`, `ring0Passed: boolean`, `ring1Passed: boolean | null`, `ring2Passed: boolean | null`, `iterations: number`.
   - A `PipelineStats` interface with: `documentsValidated: { spec: number, impl: number, task: number }`, `crossLevelChecksPassed: number`, `totalLlmCalls: number`, `totalTokenUsage: number`, `refinementIterations: { spec: number, impl: number, task: number }`, `escalationCount: number`.

2. Implement `runPipeline(specId: string, config?: PipelineConfig, existingImplDocs?: string[]): PipelineResult` that:
   - **Phase 1:** Load the spec document, call `refine(specPath, 'spec', config)`. If escalated, return result with `phase: 1`.
   - **Phase 2:** Call the spec-to-impl generation prompt function (from Impl Doc 2) via `callClaude()`, passing `existingImplDocs` for incremental adjustment. For each generated impl doc, write paired JSON/Markdown artifacts to `implementation/`, call `refine(implPath, 'impl', config)`. If any escalates, halt and return with `phase: 2`. On success, update the spec's JSON with `implementation_docs` and set status to `decomposed`.
   - **Phase 3:** For each validated impl doc, call the impl-to-task generation prompt function (from Impl Doc 3) via `callClaude()`. For each generated task, write paired JSON/Markdown artifacts to `tasks/`, call `refine(taskPath, 'task', config)`. If any escalates, halt with `phase: 3`. Run cross-task invariants (CL-T01 through CL-T05). On success, update each impl doc's JSON with `atomic_tasks` and set status to `decomposed`.
   - **Phase 4:** Run all cross-level invariants: CL-S01 through CL-S04 (from Impl Doc 1), CL-T01 through CL-T05 (from Impl Doc 2), CL-F01 and CL-F02 (from Impl Doc 3). Collect results. Return final `PipelineResult` with `phase: 4`.

3. Implement `onSpecChange(specId: string, config?: PipelineConfig): PipelineResult` that:
   - Reads the spec JSON, increments `version`.
   - Loads all downstream impl docs, reverts their status to `'draft'`.
   - Sets any `pending` or `running` execution records to `'abandoned'`.
   - Collects existing impl doc IDs as `existingImplDocs`.
   - Calls `runPipeline(specId, config, existingImplDocs)`.

4. Import `refine` from `./refine.ts`, `callClaude` from `../llm/claude-cli.ts`, per-level generation prompts and cross-level validators from Impl Doc modules, and file I/O utilities from Node.js `fs`.

## Constraints

- The four phases must execute sequentially; do not parallelize across phases.
- Halt immediately at any phase where escalation occurs — do not proceed to the next phase.
- Do not inline any validation or generation logic; delegate to imported functions.
- Do not implement CLI argument parsing; this module is a library consumed by CLI entry points.
- When updating parent documents after decomposition, preserve all existing fields — only add/update `implementation_docs`/`atomic_tasks` and `status`.
