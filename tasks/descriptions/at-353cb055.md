# at-353cb055: Pipeline: Refinement Loop

## Objective

Implement the refinement loop that drives a single document through iterative validation and automated fixes until it either passes all three rings (promotion) or reaches a convergence plateau or iteration limit (escalation). After this task, the orchestrator can call `refine()` for any document at any level and receive a definitive promoted/escalated result.

## Context

All dependencies are in place: the Claude CLI wrapper (at-08d20126), Ring 1/2 runners (at-d5c5627d), fix functions (at-2fda3e72), and convergence/escalation (at-e0ffdd40). The refinement loop implements the `refine()` pseudocode from docs/04-validation-pipeline.md almost verbatim.

Critical invariant: when any ring fails and a fix is applied, validation restarts from Ring 0. This is because a semantic fix may break structure and a quality fix may introduce semantic contradictions (REQ-06).

The loop maintains `previous_ring1_issues` and `previous_ring2_issues` as separate local variables to detect convergence independently per ring (REQ-08). Convergence is checked only at Ring 1 and Ring 2 (not Ring 0, which is deterministic).

Three termination conditions (REQ-09):
1. All three rings pass — return `{ promoted: true }`.
2. Convergence detected at Ring 1 or Ring 2 — escalate with reason `'convergence'`.
3. Max iterations reached — escalate with reason `'max_iterations'`.

## Approach

1. Create `src/pipeline/refine.ts` exporting:
   - A `RefinementResult` type: `{ promoted: true } | { escalated: true; report: string }` where `report` is the file path to the escalation report.
   - A `refine(documentPath: string, level: 'spec' | 'impl' | 'task', config: PipelineConfig): RefinementResult` function.

2. Implement the `refine()` function body:
   - Initialize `iteration = 0`, `previous_ring1_issues: IssuePair[] = []`, `previous_ring2_issues: IssuePair[] = []`, `history: IterationRecord[] = []`.
   - Enter a `while (iteration < config.refinement.max_iterations)` loop.
   - Read the current document content from `documentPath`.
   - **Ring 0:** Call the per-level Ring 0 validator (imported from Impl Docs 1/2/3 based on `level`). If Ring 0 fails, call `fixStructural()`, write the fixed content back to `documentPath`, record the iteration in `history`, increment `iteration`, continue.
   - **Ring 1:** Call `runRing1Check()` for each Ring 1 rule applicable to the document level. If any check fails, extract `(rule, reference)` pairs from the failures, call `checkConvergence()` against `previous_ring1_issues`. If converged, generate escalation report and return `{ escalated: true, report }`. Otherwise, update `previous_ring1_issues`, call `fixSemantic()`, write back, record, increment, continue.
   - **Ring 2:** Call `runRing2Check()` for each Ring 2 rule applicable to the document level. If any check fails, extract `(rule, reference)` pairs, call `checkConvergence()` against `previous_ring2_issues`. If converged, escalate. Otherwise, update `previous_ring2_issues`, call `fixQuality()`, write back, record, increment, continue.
   - If all three rings pass, return `{ promoted: true }`.
   - After loop exits (max iterations), generate escalation report with reason `'max_iterations'` and return `{ escalated: true, report }`.

3. Import per-level Ring 0 validators from Impl Doc modules, Ring 1/2 runners from `../llm/ring1.ts` and `../llm/ring2.ts`, fix functions from `../llm/fix.ts`, convergence from `./convergence.ts`, escalation from `./escalation.ts`.

## Constraints

- Validation must restart from Ring 0 after every fix application — no skipping ahead.
- `previous_ring1_issues` and `previous_ring2_issues` must be tracked separately; do not merge them into a single collection.
- Do not implement multi-document orchestration; this module handles exactly one document.
- Do not modify the document in memory only — write fixes back to the file path so that the document snapshot in escalation reports reflects the latest state.
