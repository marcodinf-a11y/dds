# at-e27804a5: Spec Schema: JSON Schema and TypeScript Types

## Objective

Define the foundational data model for specification documents by creating the JSON Schema (Draft-07) for SpecificationDefinition and the TypeScript interfaces for definitions and validation results. After this task, the type system and schema are available for all downstream validators and prompt templates to import.

## Context

No codebase exists yet. This task creates the first source files in the project. The planned directory structure places JSON Schema files under `src/schemas/`, TypeScript type definitions under `src/types/`.

The SpecificationDefinition schema must enforce the following fields and constraints from the spec:
- `id`: string matching `spec-[0-9a-f]{8}` (REQ-02)
- `title`: non-empty string
- `description`: string matching `spec-[0-9a-f]{8}\.md` (REQ-03)
- `status`: enum of `draft`, `validated`, `decomposed` (REQ-04)
- `version`: integer, minimum 1
- `implementation_docs`: optional array of strings matching `impl-[0-9a-f]{8}`
- `related_specs`: optional array of strings matching `spec-[0-9a-f]{8}`

The TypeScript interfaces must mirror the schema. The result types are shared across all document levels:
- `Ring0Result`: contains `results: Array<{ rule: string; passed: boolean; message?: string }>` and a computed `valid: boolean`
- `Ring1Result`: contains `results: Array<{ check: string; verdict: string; issues: string[] }>`
- `Ring2Result`: contains `results: Array<{ check: string; dimension: string; verdict: string; evidence: string; summary: string }>`

Naming conventions: kebab-case files, PascalCase interfaces/types, camelCase functions, UPPER_SNAKE_CASE constants.

## Approach

1. Create `src/schemas/spec.schema.json` with `$schema` set to `http://json-schema.org/draft-07/schema#`, `title` set to `SpecificationDefinition`, and `type` set to `object`. Define `required` as `["id", "title", "description", "status", "version"]`. Add `additionalProperties: false`. Define each property with the type, pattern, and enum constraints listed in the Context section.

2. Create `src/types/definitions.ts`. Export a `SpecDefinition` interface with fields `id: string`, `title: string`, `description: string`, `status: 'draft' | 'validated' | 'decomposed'`, `version: number`, `implementation_docs?: string[]`, `related_specs?: string[]`. This file will be extended by sibling impl docs to add `ImplDefinition` and `TaskDefinition`.

3. Create `src/types/results.ts`. Export interfaces `Ring0RuleResult` with fields `rule: string`, `passed: boolean`, `message?: string`. Export `Ring0Result` with fields `results: Ring0RuleResult[]` and `valid: boolean`. Export `Ring1CheckResult` with fields `check: string`, `verdict: string`, `issues: string[]`. Export `Ring1Result` with field `results: Ring1CheckResult[]`. Export `Ring2CheckResult` with fields `check: string`, `dimension: string`, `verdict: string`, `evidence: string`, `summary: string`. Export `Ring2Result` with field `results: Ring2CheckResult[]`.

## Constraints

- Do not add any runtime dependencies beyond what will be in `package.json` (this task does not add `ajv`; that is done when the Ring 0 validator is implemented).
- Do not create validator functions; only type definitions and the schema file.
- Do not modify files outside the declared scope.
- Use `interface` (not `type` alias) for all exported types to allow declaration merging in later impl docs.
