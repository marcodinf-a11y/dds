# at-17bc59d3: Spec Schema: Cross-Level Validators

## Objective

Implement the cross-level invariant checker for Spec-to-Implementation Document relationships (CL-S01 through CL-S04). After this task, the system can detect bidirectional reference inconsistencies, missing impl docs for decomposed specs, incomplete functional area coverage, and version/status mismatches between specs and their downstream impl docs.

## Context

Cross-level validators live in `src/validators/cross-level/spec-impl.ts`, separate from the per-document Ring 0 validator. They require data from both spec and impl doc definitions, which is a different data access pattern than single-document validation.

The function imports:
- `SpecDefinition` from `src/types/definitions.ts` (created in at-e27804a5)
- `Ring0Result`, `Ring0RuleResult` from `src/types/results.ts` (created in at-e27804a5)
- `extractHeadings` from `src/parsers/markdown.ts` (created in at-73d6ad76)

Note: The `ImplDefinition` type does not yet exist (it will be created by Impl Doc 2, impl-3c8d5e0a). For now, define a minimal inline interface or use a structural type with the fields needed: `{ id: string; spec_sections: string[]; status: string }`. This avoids a circular dependency on a type that does not exist yet.

The four cross-level invariants:
- **CL-S01:** Bidirectional consistency. Every impl doc's `spec_sections` entries reference a spec whose `implementation_docs` list contains that impl doc's ID, and vice versa. Every spec ID in a spec's `implementation_docs` must correspond to an impl doc whose `spec_sections` references back to that spec.
- **CL-S02:** Decomposed specs must have impl docs. Every spec with `status: decomposed` has at least one entry in `implementation_docs`.
- **CL-S03:** Functional area coverage. The union of all `spec_sections` entries across a spec's impl docs covers every H3 heading (functional area) under the Functional Requirements H2 section. Uses `extractHeadings` to find H3s and compares their slugs against the `spec_sections` heading fragments.
- **CL-S04:** Version/status consistency. When a spec's version is greater than 1 (implying it has been incremented), all downstream impl docs must have `status: draft`. This detects stale impl docs that were not reverted after a spec version bump.

All four invariants are deterministic (Ring 0 complexity), no LLM involvement.

## Approach

1. In `src/validators/cross-level/spec-impl.ts`, define a minimal interface `ImplDocMinimal` with fields `id: string`, `spec_sections: string[]`, `status: string` to represent impl doc data without depending on the not-yet-created ImplDefinition type.

2. Define a helper function `checkRule(rule: string, passed: boolean, message?: string): Ring0RuleResult`.

3. Implement and export `validateSpecImplCrossLevel(spec: SpecDefinition, implDocs: ImplDocMinimal[], specMarkdown: string): Ring0Result`. The function builds an array of `Ring0RuleResult` entries:

4. **CL-S01 check:** For each impl doc, extract the spec IDs from its `spec_sections` entries (the `spec-XXXXXXXX` prefix before the `#`). Check that `spec.id` appears among them if the impl doc is in `spec.implementation_docs`. Conversely, for each ID in `spec.implementation_docs`, verify that the corresponding impl doc's `spec_sections` contains at least one entry starting with `spec.id#`. Report any mismatches.

5. **CL-S02 check:** If `spec.status === 'decomposed'`, verify `spec.implementation_docs` is defined and has length > 0. Pass trivially if status is `draft` or `validated`.

6. **CL-S03 check:** Call `extractHeadings(specMarkdown)` to find all H3 headings that appear between the "Functional Requirements" H2 and the next H2. Collect their slugs prefixed with `spec.id#` (e.g., `spec-fa3a90b8#user-authentication`). Compute the union of all `spec_sections` entries across the provided impl docs. Check that every functional area slug is covered.

7. **CL-S04 check:** If `spec.version > 1`, check that every impl doc in the provided list has `status === 'draft'`. Report any impl doc with non-draft status.

8. Set `Ring0Result.valid` to true only if all results pass. Return the result.

## Constraints

- Do not perform file I/O; all data is passed as function parameters.
- Do not call any LLM or external service.
- Do not import the full ImplDefinition type (it does not exist yet); use the minimal inline interface.
- Do not modify files outside the declared scope.
- All four invariants must be deterministic checks with no probabilistic components.
