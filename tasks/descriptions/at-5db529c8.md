# at-5db529c8: Impl Schema: Cross-Level Validators

## Objective

After this task, the project has a cross-level invariant checker for Implementation Document to Atomic Task relationships, implementing CL-T01 through CL-T05 as deterministic pure functions. This enables the validation pipeline to verify structural consistency across the impl-doc and atomic-task hierarchy levels.

## Context

**File:** `src/validators/cross-level/impl-task.ts`.

**Pattern:** Cross-level validators are deterministic (Ring 0 complexity), take pre-loaded data as parameters (no file I/O), and return structured results. They are separate from the per-document Ring 0 validators because they operate across document boundaries.

**The five invariants:**

- **CL-T01 (Bidirectional consistency):** Every atomic task's `parent` field references an impl doc that lists that task in its `atomic_tasks` array, and vice versa. For every task T with `parent: impl-X`, `impl-X.atomic_tasks` must contain `T.id`. For every impl doc I listing task `T.id` in `atomic_tasks`, `T.parent` must equal `I.id`.

- **CL-T02 (Decomposed requires tasks):** Every impl doc with `status: decomposed` has at least one atomic task in its `atomic_tasks` array that actually exists as a task definition.

- **CL-T03 (Module containment):** For every impl doc I, every module in `I.modules` appears in at least one child task's `scope.modules` (full coverage). Conversely, every child task's `scope.modules` is a subset of `I.modules` (no boundary violations).

- **CL-T04 (Traceability coverage):** The union of all `context_refs` across an impl doc's atomic tasks covers all entries in the impl doc's `spec_sections`. Every spec section referenced by the impl doc must be traceable through at least one child task.

- **CL-T05 (Dependency ordering consistency):** If impl doc A depends on impl doc B (via `dependencies`), then the task dependency graph must be consistent: no task from B may be blocked_by a task from A. The impl-level ordering must not contradict the task-level ordering.

**Types consumed:**
- `ImplDefinition` from `src/types/definitions.ts`.
- Task definition type (minimal shape: `{ id: string; parent: string; scope: { modules: string[] }; context_refs: string[]; blocked_by: string[] }`). The full `AtomicTaskDefinition` type may not yet exist (impl-9f4b1c7d). Use a local interface or the minimal shape.
- `buildAdjacencyList` from `src/parsers/graph.ts` may be useful for CL-T05.

## Approach

1. Create `src/validators/cross-level/impl-task.ts`.

2. Define a local `TaskDefinitionMinimal` interface capturing the fields needed: `id`, `parent`, `scope.modules`, `context_refs`, `blocked_by`, `blocks`. This avoids a hard dependency on impl-9f4b1c7d's type.

3. Define and export the main function `validateImplTaskCrossLevel(implDocs: ImplDefinition[], tasks: TaskDefinitionMinimal[]): CrossLevelResult`. The `CrossLevelResult` type should contain an array of check results, each with `rule`, `pass`, and `message` fields (consistent with Ring0Check pattern).

4. Implement CL-T01: For each task, verify its parent impl doc lists it. For each impl doc, verify all listed tasks reference it as parent. Report mismatches.

5. Implement CL-T02: For each impl doc with `status: decomposed`, verify at least one task in `atomic_tasks` exists in the tasks array.

6. Implement CL-T03: For each impl doc, compute the set of modules covered by child tasks. Verify full coverage (every impl module appears in at least one child) and no violations (every child module is in the impl's modules). Report gaps and violations.

7. Implement CL-T04: For each impl doc, compute the union of `context_refs` across all child tasks. Verify every entry in `spec_sections` is covered. Report uncovered sections.

8. Implement CL-T05: For each impl doc dependency edge (A depends on B), verify no task from B has a `blocked_by` referencing a task from A. This would mean B's work depends on A's work, contradicting A's declaration that it depends on B.

9. Return `{ checks, pass: checks.every(c => c.pass) }`.

10. Verify with `npx tsc --noEmit`.

## Constraints

- Do not perform any file I/O. All data comes through function parameters.
- Do not call any LLM or external service. All checks must be deterministic.
- Do not import `AtomicTaskDefinition` from impl-9f4b1c7d (it may not exist yet). Use a local minimal interface.
- Do not modify files outside `src/validators/cross-level/impl-task.ts`.
- Each CL-T invariant must produce at least one check entry in the result.
