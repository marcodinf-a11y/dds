# at-cc72052d: Impl Schema: Generation Prompt Templates

## Objective

After this task, the project has a generation prompt module that defines the system prompt and user prompt template for decomposing a validated specification into implementation documents. The pipeline engine uses these prompts to invoke the LLM for impl doc generation.

## Context

**File:** `src/validators/impl/generation.ts`.

**Pattern from impl-7e2a9f1b:** Generation prompts follow the same export pattern as validation prompts. The system prompt is a constant string or function defining the LLM's role and rules. The user prompt template is a function accepting document content and context parameters, returning a formatted prompt string.

**What the generation prompt must instruct:**
- The LLM decomposes a validated spec into implementation documents.
- Each impl doc is a paired JSON definition + Markdown description.
- The JSON must conform to the ImplementationDefinition schema (impl.schema.json).
- The Markdown must follow the 7-H2-section template (Objective, Background, Requirements, Design Decisions, Out of Scope, Dependencies, Decomposition Notes) with 3 H3 subsections under Decomposition Notes.
- The LLM must assign fresh `impl-XXXXXXXX` IDs.
- Dependencies between impl docs must be declared and acyclic.
- Each requirement must include a `(from spec-XXXXXXXX#heading-slug)` traceability reference.
- The `spec_sections` field must reference actual spec headings.
- The `modules` field must partition the system into logical modules.

**Parameters for the user prompt template:**
- `spec_json`: The spec definition JSON.
- `spec_markdown`: The spec description Markdown.
- `codebase_context`: Description of existing codebase state.
- `build_command`, `test_command`, `lint_command`: Project build/test/lint commands.

## Approach

1. Create `src/validators/impl/generation.ts`.

2. Define and export `IMPL_GENERATION_SYSTEM_PROMPT` as a string constant. The system prompt must:
   - Define the LLM's role as an implementation document generator for DDS.
   - List the rules for producing valid impl doc pairs (ID format, required fields, Markdown template structure, dependency acyclicity, REQ-XX format with traceability).
   - Specify the output format: for each impl doc, a JSON definition in a ```json block followed by the full Markdown description.
   - Instruct separation of impl docs with horizontal rules.
   - Instruct output in dependency order (dependencies before dependents).

3. Define and export `buildImplGenerationPrompt(params: ImplGenerationParams): string` where `ImplGenerationParams` is an interface with fields: `spec_json: string`, `spec_markdown: string`, `codebase_context: string`, `build_command: string`, `test_command: string`, `lint_command: string`. The function returns the formatted user prompt string incorporating all parameters.

4. Export the `ImplGenerationParams` interface.

5. Verify with `npx tsc --noEmit`.

## Constraints

- Do not implement prompt execution or LLM invocation -- that belongs to impl-5a0e3d8f.
- Do not import runtime dependencies beyond TypeScript types.
- The system prompt must be a static string constant, not dynamically generated.
- The user prompt template function must be a pure function with no side effects.
- Do not reference the atomic task schema or task-level concerns -- this prompt generates impl docs, not tasks.
