# at-72a660c7: Task Schema: Generation Prompt Templates

## Objective

Define the system prompt and user prompt template for generating atomic tasks from a validated implementation document. After this task, the pipeline can invoke an LLM to decompose implementation documents into sets of paired task definition and description artifacts.

## Context

The canonical generation prompt is defined in `docs/03-atomic-task-schema.md` under "Generation Prompt: Creating Atomic Tasks from an Implementation Document". It consists of two parts:

1. **System prompt** — instructs the LLM on the rules for atomic task generation: fresh ID generation, dependency symmetry, scope exhaustiveness, criterion type requirements, approach specificity, context_refs tracing, 3-8 task target, execution order output.

2. **User prompt template** — provides the implementation document JSON and Markdown, relevant spec sections, codebase context, and build/test/lint commands as interpolation parameters.

The generation prompt pattern follows impl-7e2a9f1b (spec-to-impl-doc generation) and impl-3c8d5e0a (impl-doc generation prompts). Each level has a `generation.ts` file that exports functions for building the system and user prompts.

The generated output format is: for each task, a JSON definition in a code block followed by the full Markdown description, separated by horizontal rules, in execution order.

## Approach

1. Create `src/validators/task/generation.ts`. Import `TaskDefinition` and `ImplDefinition` from `src/types/definitions.ts`.

2. Export `buildTaskGenerationSystemPrompt(): string` — returns the system prompt matching the canonical template from docs/03-atomic-task-schema.md. The prompt covers all generation rules:
   - Generate fresh at- and ac- IDs using 8 random hex chars.
   - Set parent to the implementation document's ID.
   - Follow decomposition notes (suggested boundaries, ordering rationale, constraints).
   - Maintain dependency symmetry across all blocked_by/blocks fields.
   - scope.files must be exhaustive (harness enforces).
   - scope.modules must be a subset of parent impl doc's modules.
   - Every task must have at least one acceptance criterion.
   - Include build criterion for every task.
   - Include test criterion for tasks adding/modifying tests.
   - Include review criterion where pattern adherence matters.
   - Approach section must be step-by-step with concrete file/class/method references.
   - Context section must front-load all needed knowledge.
   - context_refs must trace to spec sections.
   - Aim for 3-8 tasks, each completable in a single agent session.
   - Output in execution order.

3. Export `buildTaskGenerationUserPrompt(params: TaskGenerationParams): string` — accepts a parameter object with fields: `implJson: string`, `implMarkdown: string`, `specSectionsContent: string`, `codebaseContext: string`, `buildCommand: string`, `testCommand: string`, `lintCommand: string`. Interpolates these into the user prompt template.

4. Define and export the `TaskGenerationParams` interface for the user prompt function parameters.

## Constraints

- Do not modify any files outside src/validators/task/generation.ts.
- Do not implement prompt execution or LLM API calls — template builders only.
- System prompt must include all rules from the canonical template without omission.
- User prompt must accept all six interpolation parameters from the canonical template.
- Do not hardcode project-specific values (build commands, IDs) in the prompts.
