# at-9958228c: Task Schema: Full-Stack Traceability Validators

## Objective

Implement the full-stack traceability invariant checks CL-F01 (top-down: every spec requirement traces to at least one task acceptance criterion) and CL-F02 (bottom-up: every task acceptance criterion traces back to at least one spec requirement). After this task, the system can verify end-to-end reference chains across all three document levels.

## Context

Full-stack traceability is unique to the task level because it is the terminus of the decomposition chain: specification -> implementation document -> atomic task -> acceptance criterion. The two invariants ensure no requirements are lost (CL-F01) and no orphan criteria exist (CL-F02).

The chain resolution works as follows:
1. **Spec level:** Each spec defines functional requirements (FR-XX) and non-functional requirements (NFR-XX) under heading slugs.
2. **Impl doc level:** Each impl doc REQ-XX has a `(from spec-XXXXXXXX#heading-slug)` annotation tracing back to a spec section. The impl doc's `spec_sections` array lists which spec sections it covers.
3. **Task level:** Each task's `context_refs` array contains `spec-XXXXXXXX#heading-slug` references. Each task has `acceptance_criteria` entries.

CL-F01 (top-down): For every spec FR-XX and NFR-XX, there must exist at least one impl doc REQ-XX referencing that spec section, and at least one task with a context_ref to that spec section and at least one acceptance criterion.

CL-F02 (bottom-up): For every task acceptance criterion, the task's context_refs must trace through the parent impl doc's spec_sections to at least one spec FR-XX or NFR-XX.

Both checks are purely structural and deterministic (REQ-20) — no LLM involvement.

The types needed are `SpecDefinition`, `ImplDefinition`, and `TaskDefinition` from `src/types/definitions.ts`. The spec definition includes section information; the impl doc definition includes `spec_sections` and requirement annotations; the task definition includes `context_refs` and `acceptance_criteria`.

Cross-level validators for spec-level (CL-S) are in impl-7e2a9f1b and for impl-doc-level (CL-T) are in impl-3c8d5e0a. The full-stack validators are in this impl doc because they terminate at the task/acceptance-criterion level.

## Approach

1. Create `src/validators/cross-level/full-stack.ts`. Import `SpecDefinition`, `ImplDefinition`, `TaskDefinition` from `src/types/definitions.ts`.

2. Define a `FullStackTraceabilityContext` interface containing: `specs: SpecDefinition[]`, `implDocs: ImplDefinition[]`, `tasks: TaskDefinition[]`, `implMarkdowns: Map<string, string>` (for extracting REQ-XX annotations with `from` references).

3. Implement a helper function `buildForwardReferenceMap(context: FullStackTraceabilityContext)` that:
   - Extracts all spec requirement heading slugs (FR-XX, NFR-XX sections).
   - For each, finds impl doc REQs that reference it via `(from spec-XXXXXXXX#heading-slug)`.
   - For each impl doc REQ, finds tasks whose `context_refs` include the relevant spec section.
   - Returns a map from spec requirement slug to the set of task acceptance criterion IDs that cover it.

4. Implement a helper function `buildBackwardReferenceMap(context: FullStackTraceabilityContext)` that:
   - For each task acceptance criterion, follows the task's `context_refs` to spec sections.
   - Verifies each spec section reference exists in the parent impl doc's `spec_sections`.
   - Returns a map from acceptance criterion ID to the set of spec requirement slugs it traces to.

5. Implement and export `validateCLF01(context: FullStackTraceabilityContext): CrossLevelResult` — iterates the forward reference map and reports any spec requirement with an empty coverage set (no task acceptance criterion covers it).

6. Implement and export `validateCLF02(context: FullStackTraceabilityContext): CrossLevelResult` — iterates the backward reference map and reports any acceptance criterion with an empty trace-back set (orphan criterion).

7. Export `validateFullStackTraceability(context: FullStackTraceabilityContext): CrossLevelResult` — convenience function that runs both CL-F01 and CL-F02 and combines results.

## Constraints

- Do not modify any files outside src/validators/cross-level/full-stack.ts.
- Checks must be purely structural and deterministic — no LLM calls.
- Do not hardcode any document IDs, spec section names, or requirement identifiers.
- Do not duplicate logic from CL-S (spec-level) or CL-T (impl-doc-level) cross-level validators — this module focuses only on the full-stack (three-level) chain.
- The function must handle the case where specs, impl docs, or tasks arrays are empty (returning appropriate failures rather than crashing).
