# at-7149508d: Task Schema: Ring 0 Unit Tests and Fixtures

## Objective

Create comprehensive unit tests and fixture files for all task-level Ring 0 validation rules (R0-T01 through R0-T34) and full-stack traceability invariants (CL-F01, CL-F02). After this task, every structural validation rule for tasks and execution records has automated test coverage with both positive and negative test cases.

## Context

The test structure follows the pattern established by impl-7e2a9f1b and impl-3c8d5e0a: unit tests live in `tests/unit/ring0/` and fixtures live in `tests/fixtures/`. Tests use vitest with `describe`/`it`/`expect` blocks organized by rule groups.

The validators under test are:
- `validateTaskRing0()` from `src/validators/task/ring0.ts` — 24 rules (R0-T01 through R0-T14, R0-T20 through R0-T24)
- `validateExecutionRecord()` from `src/validators/task/ring0.ts` — 5 rules (R0-T30 through R0-T34)
- `validateCLF01()` and `validateCLF02()` from `src/validators/cross-level/full-stack.ts`

Key test scenarios that are unique to the task level:
- **Dependency symmetry (R0-T07):** Requires loading sibling tasks. Test cases: valid symmetric pair, A.blocks has B but B.blocked_by missing A, A.blocked_by has C but C.blocks missing A, multiple dependencies all symmetric.
- **Acyclicity (R0-T08):** Test cases: valid DAG (linear chain), direct cycle (A blocks B, B blocks A), transitive cycle (A->B->C->A), self-reference caught by R0-T14 before R0-T08.
- **Criterion type discrimination (R0-T10/R0-T11):** Test cases: test criterion with verify (pass), review criterion with rubric (pass), test criterion missing verify (fail), review criterion missing rubric (fail), test criterion with rubric instead of verify (fail).
- **Execution record sequential runs (R0-T32):** Test cases: first run (run=1, no prior records), sequential run (run=2 with existing run=1), gap in runs (run=3 with only run=1 existing).
- **Full-stack traceability:** Test cases: complete chain from spec FR through impl REQ to task AC, spec FR with no impl REQ covering it (CL-F01 fail), orphan acceptance criterion with no spec trace (CL-F02 fail).

Fixtures should be separate files loaded by the tests, not inlined large objects. Valid fixtures provide a baseline; invalid fixtures each target a specific rule violation.

## Approach

1. Create `tests/fixtures/tasks/valid-task.json` — a well-formed task definition with all required fields, valid ID formats, multiple acceptance criteria of different types (at least one test, one build, one review), non-empty scope.files and scope.modules, non-empty context_refs.

2. Create `tests/fixtures/tasks/valid-task.md` — a well-formed task description matching the valid-task.json ID, with H1 `# {id}: {title}` and exactly five non-empty H2 sections (Objective, Context, Approach, Constraints, References).

3. Create `tests/fixtures/tasks/valid-execution-record.json` — a well-formed execution record referencing valid-task.json's task_id, run 1, status completed, with criterion_results matching the task's acceptance_criteria IDs.

4. Create invalid fixture files in `tests/fixtures/tasks/invalid/`:
   - `schema-violation.json` — missing required field (e.g., no `scope`).
   - `bad-id-format.json` — ID not matching `at-[0-9a-f]{8}` pattern.
   - `asymmetric-blocks.json` + `asymmetric-blocks-sibling.json` — A.blocks has B but B.blocked_by does not have A.
   - `cyclic-deps-a.json` + `cyclic-deps-b.json` + `cyclic-deps-c.json` — three tasks forming A->B->C->A cycle.
   - `duplicate-criterion-ids.json` — two acceptance criteria with the same ac- ID.
   - `missing-verify.json` — test criterion without verify field.
   - `missing-rubric.json` — review criterion without rubric field.
   - `bad-markdown.md` — wrong number of H2 sections or wrong order.
   - `self-reference.json` — task with own ID in blocked_by.
   - `bad-execution-record.json` — execution record with non-sequential run number.
   - `orphan-criterion-record.json` — execution record with criterion_id not in task.

5. Create `tests/unit/ring0/task-validator.test.ts`:
   - Import `validateTaskRing0`, `validateExecutionRecord` from `src/validators/task/ring0.ts`.
   - Import `validateCLF01`, `validateCLF02` from `src/validators/cross-level/full-stack.ts`.
   - Load fixtures using `fs.readFileSync` or vitest's fixture loading.

6. Add `describe('R0-T01 through R0-T14: Task Definition Structural Checks')` block:
   - `it('passes for a valid task definition')` — load valid-task.json with valid context.
   - `it('R0-T01: fails for schema violation')` — load schema-violation.json.
   - `it('R0-T02: fails for duplicate task ID')` — valid task with ID already in existingTaskIds.
   - `it('R0-T05/R0-T06: fails for unknown dependency reference')` — task referencing non-existent sibling.
   - `it('R0-T07: fails for asymmetric blocks')` — load asymmetric-blocks fixtures with sibling context.
   - `it('R0-T08: fails for cyclic dependencies')` — load cyclic-deps fixtures.
   - `it('R0-T09: fails for duplicate criterion IDs')` — load duplicate-criterion-ids.json.
   - `it('R0-T10: fails when test criterion missing verify')` — load missing-verify.json.
   - `it('R0-T11: fails when review criterion missing rubric')` — load missing-rubric.json.
   - `it('R0-T12: fails when scope.files is empty')` — construct task with empty files array.
   - `it('R0-T14: fails for self-reference')` — load self-reference.json.

7. Add `describe('R0-T20 through R0-T24: Task Description Markdown Checks')` block:
   - `it('passes for valid markdown')` — load valid-task.md.
   - `it('R0-T20: fails when H1 does not match pattern')` — markdown with wrong H1 format.
   - `it('R0-T21: fails with wrong number of H2 sections')` — load bad-markdown.md.
   - `it('R0-T23: fails when H2 section is empty')` — markdown with empty Approach section.
   - `it('R0-T24: fails when H1 id does not match JSON id')` — mismatched IDs.

8. Add `describe('R0-T30 through R0-T34: Execution Record Checks')` block:
   - `it('passes for a valid execution record')` — load valid-execution-record.json.
   - `it('R0-T31: fails when task_id references non-existent task')` — bad task_id.
   - `it('R0-T32: fails for non-sequential run number')` — load bad-execution-record.json.
   - `it('R0-T33: fails for invalid criterion_id')` — load orphan-criterion-record.json.
   - `it('R0-T34: fails for duplicate criterion_id in results')` — construct record with duplicates.

9. Add `describe('CL-F01 and CL-F02: Full-Stack Traceability')` block:
   - `it('CL-F01: passes when all spec requirements have full chain')` — construct complete chain fixtures.
   - `it('CL-F01: fails when spec requirement has no task coverage')` — spec FR with no task.
   - `it('CL-F02: passes when all criteria trace back to spec')` — same complete chain.
   - `it('CL-F02: fails for orphan acceptance criterion')` — criterion with no spec trace.

## Constraints

- Do not modify any source files (src/) — this task creates only test and fixture files.
- Fixtures must be in `tests/fixtures/tasks/` and loaded by tests, not inlined as large objects.
- Each invalid fixture must target exactly one rule violation to keep tests isolated.
- Tests must use vitest (describe/it/expect) — no other test framework.
- Do not add test dependencies beyond vitest (already in the project).
