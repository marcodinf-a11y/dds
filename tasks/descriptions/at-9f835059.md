# at-9f835059: Impl Schema: Ring 1 and Ring 2 Prompt Templates

## Objective

After this task, the project has prompt template functions for all six Ring 1 semantic checks (R1-I10 through R1-I15) and all six Ring 2 quality rubrics (R2-I10 through R2-I15) for implementation documents. These templates are consumed by the pipeline engine (impl-5a0e3d8f) to invoke LLM-based validation.

## Context

**Files:** `src/validators/impl/ring1.ts` and `src/validators/impl/ring2.ts`.

**Pattern from impl-7e2a9f1b:** Prompt templates are exported functions with signature `(documentContent: string, additionalContext?: string) => string`. Each returns a complete prompt string that the pipeline engine passes to the `claude` CLI. The `additionalContext` parameter handles cases like R1-I15 (cross-implementation contradiction), which needs sibling impl doc content.

**Ring 1 semantic checks (each asks exactly one question, expects JSON with check/verdict/issues):**
- R1-I10: Spec coverage -- do impl doc requirements cover all spec section content?
- R1-I11: Out-of-scope consistency -- are out-of-scope items truly outside spec scope?
- R1-I12: Design decision coherence -- are design decisions internally consistent and well-justified?
- R1-I13: Dependency completeness -- are all required dependencies declared?
- R1-I14: Decomposition coverage (conditional on `decomposed` status) -- do suggested tasks cover all requirements?
- R1-I15: Cross-implementation contradiction -- does this impl doc contradict sibling impl docs?

**Ring 2 quality rubrics (each produces JSON with check/dimension/verdict/evidence/summary):**
- R2-I10: Decomposability -- can this impl doc be cleanly decomposed into atomic tasks?
- R2-I11: Requirement testability -- are requirements specific enough to verify?
- R2-I12: Background sufficiency -- does the background provide enough context for implementation?
- R2-I13: Design decision completeness -- are all significant decisions documented with alternatives?
- R2-I14: Boundary clarity -- are in-scope/out-of-scope boundaries unambiguous?
- R2-I15: Decomposition notes quality -- are decomposition notes specific and actionable?

## Approach

1. Create `src/validators/impl/ring1.ts`.

2. Export six functions, one per Ring 1 rule. Each function takes `(documentContent: string, additionalContext?: string)` and returns a prompt string. The prompt must:
   - Identify the check by its rule ID (e.g., R1-I10).
   - Ask exactly one question.
   - Provide the document content inline.
   - Instruct the LLM to respond with JSON: `{ "check": "R1-Ixx", "verdict": "pass" | "fail", "issues": [...] }`.
   - For R1-I14, include a note that the check applies only when status is `decomposed`.
   - For R1-I15, use `additionalContext` to inject sibling impl doc content.

3. Create `src/validators/impl/ring2.ts`.

4. Export six functions, one per Ring 2 rule. Each function takes `(documentContent: string, additionalContext?: string)` and returns a prompt string. The prompt must:
   - Identify the rubric by its rule ID (e.g., R2-I10).
   - Define the quality dimension being assessed.
   - Provide the document content inline.
   - Instruct the LLM to respond with JSON: `{ "check": "R2-Ixx", "dimension": "...", "verdict": "pass" | "fail", "evidence": "...", "summary": "..." }`.

5. Verify with `npx tsc --noEmit`.

## Constraints

- Do not implement prompt execution logic -- that belongs to impl-5a0e3d8f (pipeline engine).
- Do not import or call the `claude` CLI or any LLM API.
- Each function must return a single string (the prompt). No side effects.
- Do not duplicate prompt boilerplate -- use a shared helper within each file if needed, but do not create a separate shared file.
- Each Ring 1 prompt must ask exactly one question (not multiple).
- Response format instructions must be explicit in each prompt (JSON field names and types).
