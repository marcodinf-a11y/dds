# at-e1c51f43: Impl Schema: Ring 0 Validator

## Objective

After this task, the project has a Ring 0 structural validator for implementation documents that runs all 18 deterministic checks (R0-I40 through R0-I51, R0-I60 through R0-I67) and returns a typed Ring0Result. This validator enables the validation pipeline to structurally validate any impl doc before semantic or quality checks.

## Context

**File:** `src/validators/impl/ring0.ts`.

**Pattern from impl-7e2a9f1b:** Ring 0 validators are pure functions with signature `(definition, markdown, context) => Ring0Result`. The spec-level validator does not use a context parameter because spec checks are self-contained. The impl-level validator needs a context parameter (`ImplValidationContext`) because several rules require cross-document data:
- R0-I41 (ID uniqueness) needs existing impl doc IDs.
- R0-I44 (atomic_tasks reference validity) needs existing task IDs.
- R0-I45 (dependencies reference validity) needs existing impl doc IDs.
- R0-I47 (acyclicity) needs the full dependency graph across all impl docs.
- R0-I50 (parent consistency) needs task definitions to check parent references.
- R0-I51 (module containment) needs task definitions to check module subsets.

**Dependencies consumed:**
- `src/schemas/impl.schema.json` -- loaded and compiled with ajv for R0-I40.
- `src/types/definitions.ts` -- `ImplDefinition` type.
- `src/parsers/graph.ts` -- `buildAdjacencyList()` and `detectCycles()` for R0-I47.
- `src/parsers/markdown.ts` -- heading extraction for R0-I60 through R0-I67 (already exists from impl-7e2a9f1b).
- Result types from impl-7e2a9f1b (e.g., `Ring0Result`, `Ring0Check`).

**The 18 rules:**

JSON/definition checks:
- R0-I40: JSON validates against ImplementationDefinition schema (ajv).
- R0-I41: ID is unique across all impl docs (via context.existingImplIds).
- R0-I42: description field matches `impl-XXXXXXXX.md` pattern.
- R0-I43: Every spec_sections entry matches `spec-XXXXXXXX#heading-slug` format.
- R0-I44: Every atomic_tasks entry references an existing task definition (via context).
- R0-I45: Every dependencies entry references an existing impl doc (via context).
- R0-I46: No self-references in dependencies.
- R0-I47: Dependency graph is acyclic (uses graph.ts detectCycles).
- R0-I48: Status-array consistency (draft/validated = empty atomic_tasks, decomposed = non-empty).
- R0-I50: Parent consistency -- every task in atomic_tasks has parent == this impl ID (via context).
- R0-I51: Module containment -- every task's scope.modules is a subset of this impl's modules (via context).

Markdown checks:
- R0-I60: H1 matches `# {impl-id}: {title}`.
- R0-I61: Exactly 7 H2 sections in order: Objective, Background, Requirements, Design Decisions, Out of Scope, Dependencies, Decomposition Notes.
- R0-I62: No H2 section is empty.
- R0-I63: Decomposition Notes has exactly 3 H3 subsections: Suggested Task Boundaries, Ordering Rationale, Decomposition Constraints.
- R0-I64: H1 impl-id matches definition's id field.
- R0-I66: At least one REQ-XX entry in the Requirements section.
- R0-I67: Each REQ-XX entry includes a `(from spec-XXXXXXXX#heading-slug)` reference.

## Approach

1. Create `src/validators/impl/ring0.ts`.

2. Define and export the `ImplValidationContext` interface:
   ```
   interface ImplValidationContext {
     existingImplIds: string[];
     existingTaskIds: string[];
     taskDefinitions: Array<{ id: string; parent: string; scope: { modules: string[] } }>;
     dependencyGraph: Array<{ from: string; to: string }>;
   }
   ```

3. Import `ImplDefinition` from `src/types/definitions.ts`, the impl JSON schema, `buildAdjacencyList`/`detectCycles` from `src/parsers/graph.ts`, the heading extractor from `src/parsers/markdown.ts`, and result types from impl-7e2a9f1b.

4. Implement `validateImplRing0(impl: ImplDefinition, markdown: string, context: ImplValidationContext): Ring0Result`. Initialize an empty checks array. Run each rule as a separate function or inline check, pushing a `Ring0Check` object with `rule`, `pass`, and `message` fields for each.

5. Implement JSON/definition checks (R0-I40 through R0-I51):
   - R0-I40: Compile the JSON schema with ajv and validate `impl`. Report all ajv errors.
   - R0-I41: Check `impl.id` is not in `context.existingImplIds`.
   - R0-I42: Verify `impl.description` matches the regex `^impl-[0-9a-f]{8}\.md$`.
   - R0-I43: Verify each `impl.spec_sections` entry matches `^spec-[0-9a-f]{8}#[a-z0-9-]+$`.
   - R0-I44: Verify each `impl.atomic_tasks` entry exists in `context.existingTaskIds`.
   - R0-I45: Verify each `impl.dependencies` entry exists in `context.existingImplIds`.
   - R0-I46: Verify `impl.dependencies` does not contain `impl.id`.
   - R0-I47: Call `buildAdjacencyList(context.dependencyGraph)` then `detectCycles()`. Fail if cycles found.
   - R0-I48: If status is draft/validated, check `atomic_tasks.length === 0`; if decomposed, check `atomic_tasks.length > 0`.
   - R0-I50: For each task ID in `impl.atomic_tasks`, find it in `context.taskDefinitions` and verify `task.parent === impl.id`.
   - R0-I51: For each task in `context.taskDefinitions` where `task.parent === impl.id`, verify `task.scope.modules` is a subset of `impl.modules`.

6. Implement Markdown checks (R0-I60 through R0-I67):
   - R0-I60: Extract H1 from markdown and verify it matches `# {impl.id}: `.
   - R0-I61: Extract H2 headings and verify exactly 7 in order.
   - R0-I62: For each H2 section, verify it has non-whitespace content.
   - R0-I63: Within the Decomposition Notes section, extract H3 headings and verify exactly 3 in order.
   - R0-I64: Extract the impl-id from the H1 heading and compare to `impl.id`.
   - R0-I66: Search the Requirements section content for at least one `REQ-\d+` match.
   - R0-I67: For each REQ-XX entry, verify it contains `(from spec-` reference.

7. Return `{ checks, pass: checks.every(c => c.pass) }`.

8. Verify with `npx tsc --noEmit`.

## Constraints

- Do not perform any file I/O. All data comes through function parameters.
- Do not call any LLM or external service.
- Do not modify files outside `src/validators/impl/ring0.ts`.
- Each rule must produce exactly one Ring0Check entry in the result (18 total for a complete run).
- Use the same result types (`Ring0Result`, `Ring0Check`) as the spec-level validator from impl-7e2a9f1b.
