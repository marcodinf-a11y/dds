# at-639f101c: Pipeline: Unit Tests

## Objective

Implement unit tests for convergence detection and pipeline orchestration logic, mocking the Claude CLI boundary. After this task, the core pipeline logic has automated regression tests that run without invoking the real `claude` CLI.

## Context

All pipeline modules are complete (at-08d20126 through at-e6ef2a88). Tests mock at the `claude-cli.ts` module boundary — replacing `callClaude()` with predetermined responses — not at the process level. This follows the decomposition constraint that tests mock the `claude-cli.ts` boundary specifically.

The test framework is Vitest (`npx vitest run`). Tests cover two areas:
1. **Convergence detection** (`convergence.test.ts`): Pure logic tests with no mocking needed. Verifies set-overlap math across various scenarios.
2. **Pipeline logic** (`pipeline-logic.test.ts`): Tests refinement loop termination conditions and orchestration phase sequencing with mocked LLM responses.

## Approach

1. Create `tests/unit/convergence.test.ts` with the following test cases:
   - **Identical issue sets:** Both current and previous contain the same `(rule, reference)` pairs. Expect `checkConvergence()` to return `true` (overlap = 1.0, exceeds any threshold <= 1.0).
   - **Completely disjoint sets:** No overlap between current and previous. Expect `false`.
   - **Partial overlap at threshold boundary:** Construct sets where overlap ratio equals exactly the threshold (e.g., 0.7). Expect `true` (>= threshold). Construct sets just below threshold. Expect `false`.
   - **Empty previous issues:** First iteration scenario. Expect `false` (no convergence on first iteration).
   - **Empty current issues:** All issues resolved. Expect `false` (no convergence when current is empty).
   - **Single-element sets:** Edge case with one issue each, same or different.

2. Create `tests/unit/pipeline-logic.test.ts` with the following test cases:
   - **Setup:** Use `vi.mock('../../../src/llm/claude-cli.ts')` to mock `callClaude()` at the module boundary. Configure mock to return predetermined Ring 1, Ring 2, and fix responses based on call sequence.
   - **Refinement — promotion:** Mock all rings to pass on first iteration. Call `refine()`, assert it returns `{ promoted: true }`.
   - **Refinement — convergence escalation:** Mock Ring 0 to pass, Ring 1 to fail with the same issues across two iterations. Assert `refine()` returns `{ escalated: true }` with reason `'convergence'`.
   - **Refinement — max iteration escalation:** Mock Ring 0 to always fail (fix never resolves the issue). Assert `refine()` returns `{ escalated: true }` with reason `'max_iterations'` after `max_iterations` loops.
   - **Refinement — restart from Ring 0 after fix:** Mock Ring 1 to fail on first call and pass on second. Verify that Ring 0 is re-invoked after the fix (not skipped).
   - **Orchestration — four-phase happy path:** Mock all validation and generation calls to succeed. Call `runPipeline()`, assert all four phases complete and status is `'completed'`.
   - **Orchestration — halting on escalation:** Mock Phase 2 to produce an escalation. Assert `runPipeline()` returns with `phase: 2` and status `'escalated'`, and that Phase 3/4 are never reached.

3. Import `checkConvergence` from `../../src/pipeline/convergence.ts`, `refine` from `../../src/pipeline/refine.ts`, `runPipeline` from `../../src/pipeline/orchestrate.ts`.

## Constraints

- No tests may invoke the real `claude` CLI — all LLM interactions must be mocked.
- Mock at the `claude-cli.ts` module boundary using `vi.mock()`, not at the `child_process` level.
- Tests must be deterministic and not depend on file system state beyond what the test setup creates.
- Use `beforeEach` / `afterEach` to reset mocks and clean up any temporary files.
- Do not test CLI entry points (those are thin wrappers; integration testing is out of scope for this task).
